# 1장. 사용자 수에 따른 규모 확장성 
대규모 시스템 설계는 한 번에 완성되는 것이 아니라 **지속적인 확장과 개선의 과정** 



## 1.  단일서버 
웹, 앱, 데이터베이스,캐 시 등이 전부 서버 한대에서 실행된다고 하자 

사용자 요청의 처리 흐름을 살펴보면 

### DNS 조회 
- 사용자는 보통 IP 주소가 아닌 도메인 이름을 통해 서비스에 접속한다. ex) www.naoor.com
- 하지만 인터넷 통신은 실제로 IP 주소를 기반으로 이루어지기 때문에 도메인을 IP주소로 변환하는 과정이 필요하다. 
- 이 역할을 하는게 **DNS(Domain Name System)** 이다. 
- DNS 서버는 일반적으로 외부(유료) 서비스를 사용한다. (Google DNS, AWS Route53 ...)

### HTTP 요청 전달 
- DNS 조회를 통해 얻은 IP주소로 HTTP 요청이 전달된다. 
- ex) GET /users/123 
- HTTP 요청에는 다음 정보들이 표함된다. 
  - Method(GET/POST/PUT/DELETE)
  - URL 
  - Header 와 Body
  
### 서버 응답 반환 
- 서버는 요청 처리 후 클라이언트에게 **응답** 을 반환한다.
- 응답 형식은 보통 두가지로 구성된다.  
  - **HTML 응답** 
    - 웹 브라우저가 직접 렌더링할 때 사용된다. 
    - Content-Type: text/html
  - **JSON 응답**
    - API 기반 서비스에서 가장 많이 사용하는 형태이다. 
    - Content-Type: application/json
``` json
  {
  "id": 12,
  "firstName": "John",
  "lastName": "Smith",
  "address": {
    "streetAddress": "21 2nd Street",
    "city": "New York",
    "state": "NY",
    "postalCode": 10021
  },
  "phoneNumbers": [
    "212-555-1234",
    "646-555-4567"
  ]
}
```

### 클라이언트 요청 
클라이언트 요청은 두가지 단말로부터 오는데 하나는 웹 앱이고 하나는 모바일 앱이다. 
- **웹 앱** 
  - 구성요소는 서버(Java, Python)언어와 클라이언트 구현용 언어(HTML, 자바스크립트) 가있다.
  - Browser -> HTTP 요청 -> Web Server -> Database 구조로 동작한다. 
- **모바일 앱** 
  - 모바일 앱 역시 서버와 통신하기 위해 HTTP API를 사용한다. 
  - 모바일 앱은 화면은 앱 내부에서 처리하고 데이터만 JSON API로 받아온다. 



## 2.  데이터베이스
사용자가 증가하면서 하나의 서버에서 모든 기능을 처리하기 어려워진다. 
따라서 **역할별로 분리**하는 구조이다. 

대표적 방식이 웹 계층(Web Tier)과 데이터 계층(Data Tier) 분리이다. 

### 데이터베이스 종류 
데이터 저장소는 크게 두 가지 유형으로 나뉜다. 
 - 관계형 데이터베이스 (RDBMS)
 - 비관계형 데이터베이스 (NoSQL) 

### 관계형 데이터베이스 (RDBMS) 
관계형 데이터베이스는 **테이블 기반 구조로 데이터를 저장**하는 데이터베이스이다. 

- MySQL, 오라클 데이터베이스, PostgreSQL 등 
- 데이터는 테이블 형태로 저장되며 SQL을 통해 조회 및 조작한다. 

특징 
- 데이터 구조가 명확 
- 데이터 무결성 보장 
- 트랜잭션 지원 
- JOIN 연산 지원 

### 비관계형 데이터베이스 (NoSQL) 
비관계형 데이터베이스는 **테이블 기반 구조가 아닌 다양한 형태로 데이터를 저장**하는 데이터베이스이다. 
- CouchDB, Neo4j, Cassandra, HBase, Amazon DynamoDB 등
- 대부분의 NoSQL 데이터베이스는 JOIN 연산을 지원하지 않는다. 
- 대신 데이터를 비정규화 형태로 저장하는 경우가 많다. 

### NoSQL 데이터베이스 종류 
NoSQL은 크게 네 가지 유형으로 나뉜다. 

#### KEY-VALUE Store (키-값 저장소)
키와 값 형태로 데이터를 저장하는 방식 
``` json
key : user:12
value : {name:"John", age:30}
``` 
대표 시스템 
- Redis
- DynamoDB

특징 
- 조회 속도가 매우 빠름 
- 캐시 시스템으로 많이 사용됨 

#### Graph Store (그래프 저장소)
데이터 간 관계 구조를 그래프로 표현하는 데이터베이스 

대표 시스템
- Neo4j

특징
- 관계 중심 데이터 분석에 유리 
- 추천 시스템 등에 사용  

#### Column Store (칼럼 저장소)
데이터를 열 기반 구조로 저장하는 데이터베이스 

대표 시스템 
- Cassandra 
- HBase 

특징 
- 대용량 데이터 처리에 강함 
- 분산 시스템에 적합 

#### Document Store (문서저장소)
JSON 형태 문서를 저장하는 데이터베이스 
``` json
{
  "id": 12,
  "name": "John",
  "age": 30
}
``` 
대표 시스템 
- MongoDB 
- CouchDB 

특징 
- 스키마가 유연함 
- JSON 데이터 저장에 적합 

### NoSQL을 사용하는 경우 

#### 매우 낮은 응답 지연이 필요한 경우 
- 캐시 시스템, 실시간 서비스 
#### 비정형 데이터 처리 
- 데이터 구조가 고정되지 않은 경우 (로그데이터, 사용자 행동 데이터, JSON 데이터)
#### 대용량 데이터 처리 
- 빅테이터 시스템, 로그 분석 시스템 
#### 직렬화 데이터 저장 
- JSON, YAML, XML 등 문서 기반 데이터 저장이 필요한 경우 

## 3. 수직적 규모 확장 vs 수평적 규모 확장 
서비스의 사용자 수와 트래픽이 증가하면서 시스템의 처리 능력을 확장해야 한다. 
이때 시스템을 확장하는 방식은 크게 두 가지로 나뉜다. 
- 수직적 규모 확장(Vertical Scaling)
- 수평적 규모 확장(Horizontal Scaling)
  
이 두 방식은 시스템 설계에서 **성능 확장 전략의 기본 개념** 이다.

### 수직적 규모 확장(Vertical Scaling)
수직적 규모 확장은 **하나의 서버 성능을 높이는 방식**이다.
일반적으로 **Scale Up** 이라고 부른다. 

예를 들어 기존 서버의 자원을 업그레이드 하는 방식이다. 
- CPU 성능 증가 
- RAM 용량 증가 
- 디스크 성능 향상(SSD)
- 네트워크 대역폭 증가 

이 방식의 가장 큰 장점은 **구조가 단순하다는 점**이다. 
<br>서버를 추가하거나 시스템 구조를 변경할 필요 없이 **하드웨어만 업그레이드** 하면 된다. 
<br>그래서 초기 서비스나 트래픽이 적은 시스템에서 **수직적 확장이 가장 간단한 선택**이 된다. 

#### 수직적 확장의 한계 
하지만 수직적 확장에는 몇 가지 중요한 한계가 존재한다. 

#### 물리적인 확장 한계 
서버의 CPU와 메모리를 **무한히 늘릴 수는 없다**.
<br>결국 하나의 서버가 처리할 수 있는 성능에는 **물리적인 제한**이 존재한다. 

#### 장애 대응이 어렵다
수직적 확장은 **서버가 하나인 구조**를 유지하기 때문에 
<br>서버 장애가 발생하면 서비스 전체가 중단된다. 

#### 대규모 트래픽 대응이 어렵다
사용자가 매우 많아지면 하나의 서버로는 요청을 처리하기 어렵다. 
<br> 예를 들어 하나의 서버에 사용자 10만명이 몰리면 서버가 과부하 상태가 되며 응답속도지연, 요청실패, 서버 다운같은 문제가 발생할 수있다. 

### 수평적 규모 확장(Horizontal Scaling)
수평적 규모 확장은 **서버를 여러 대 추가하여 시스템 성능을 확장하는 방식**이다. 
일반적으로 **Scale Out** 이라고 부른다. 

- Web Server 1대 -> Web Server 5대 
이 방식은 여러 서버가 동시에 요청을 처리하기 때문에 **대규모 트래픽 처리에 유리**하다. 

#### 수평 확장의 핵심 요소 
수평 확장 구조에서는 **로드 밸런서(Load Balancer)** 가 중요한 역할을 한다. 
<br> 로드 밸런서는 **여러 서버로 트래픽을 분산시키는 장치** 이다.

로드 밸런서는 사용자 요청을 **여러 서버에 균항하게 분배**하여 특정 서버에 부하가 집중되는 것을 방지한다. 

#### 수평 확장의 장점 
수평 확장은 대규모 서비스에서 매우 중요한 확장 전략이다. 

#### 높은 확장성 
트래픽이 중가하면 **서버를 계속 추가할 수 있다.** 

#### 장애 대응 가능 
여러 서버가 존재하기 때문에 **한 서버가 장애가 발생해도 서비스는 계속 운영**될 수 있다. 

#### 트래픽 분산 
로드 밸런서를 통해 요청을 분산하여 **서버 부하를 줄일 수 있다.** 

## 4. 로드 밸런서 (Load Balancer)
사용자가 증가하면 웹 서버 하나로 모든 요청을 처리하기 어려워진다. 
<br> 이 문제를 해결하기 위해 여러 대의 웹 서버를 두고 트래픽을 분산시키는 구조를 사용한다. 

이때 **여러 서버로 요청을 분산시키는 역할을 하는 장치**가 바로 **로드 밸런서 (Load Balancer)** 이다. 

로드 밸런서는 **부하 분산 집합(load balancing set)** 에 속한 여러 웹 서버들에게 트래픽을 고르게 분배하는 역할을 한다. 

### Public IP 와 Private IP 
로드 밸런서를 사용하는 구조에서는 **IP 주소의 역할도 분리**된다. 

#### Public IP 
사용자가 인터넷을 통해 접속할 수 있는 주소이다. 

``` json
예시)
api.service.com
↓
Public IP
↓
Load Balancer
``` 
사용자는 **웹 서버가 아닌 로드 밸런서의 Public IP로 접속**한다. 

#### Private IP 
Private IP는 **같은 내부 네트워크에서만 사용할 수 있는 IP 주소**이다. 

특징 
- 인터넷에서 직접 접근 불가능 
- 내부 서버 간 통신에 사용 
- 보안 강화 효과 

``` json
Client
   ↓
Public IP (Load Balancer)
   ↓
Private Network
   ├ Web Server 1 (Private IP)
   ├ Web Server 2 (Private IP)
   └ Web Server 3 (Private IP)
``` 
로드 밸런서는 웹 서버들과 통신할 때 Private IP를 사용한다. 

#### 로드 밸런서의 장애 대응 (Failover)
로드 밸런서를 사용하면 **서버 장애 발생 시 자동으로 트랙픽을 다른 서버로 전달** 할 수있다. 
<br>이 방식 덕분에 서비스 전체가 중단되는 상황을 방지할 수 있다. 
<br>이 개념을 **Failover(장애 자동 복구)** 라고 한다. 

## 데이터베이스 다중화 (Database Replication) 
대규모 시스템에서는 데이터베이스 하나로 모든 요청을 처리하기 어렵다. 
특히 대부분의 서비스에서는 **읽기(Read)** 요청이 **쓰기(Write)** 요청보다 훨씬 많다.

이 문제를 해결하기 위해 **데이터베이스 다중화(Database Replication)** 구조를 사용한다. 

데이터 베이스 다중화는 **주 데이터베이스(Master)와 부 데이터베이스(Slave) 구조**로 운영된다. 

구조 예시 
``` json
Application Server
        ↓
     Master DB (Write)
        ↓
   데이터 복제
        ↓
   Slave DB 1 (Read)
   Slave DB 2 (Read)
   Slave DB 3 (Read)
``` 

### Master / Slave 구조 
데이터베이스 다중화에서는 데이터 변경 작업과 조회 작업을 분리한다. 

### Master Database 
Master DB는 **데이터 변경 작업(Write Operation)** 을 담당한다. 
- INSERT 
- UPDATE 
- DELETE 

Master DB에 저장된 데이터는 **Slave DB로 복제**된다. 

### Slave Database 
Slave DB는 **조회(Read Operation)** 을 담당한다. 
- SELECT 

Slave DB는 Master DB의 데이터를 복제하여 **읽기 전용 데이터베이스로 사용**된다. 

대부분의 서비스에서는 조회 요청이 훨씬 많기 때문에 
<br>Slave DB를 여러 개 두는 경우가 일반적이다. 

### 데이터베이스 다중화의 장점 
#### 성능 향상 
읽기 요청을 여러 Slave DB가 처리할 수  있기 때문에 
<br>데이터베이스 처리 성능이 크게 향상된다. 
``` json
Read Query
→ Slave DB 1
→ Slave DB 2
→ Slave DB 3
``` 
여러 DB에서 병렬로 쿼리를 처리할 수 있다. 

#### 안정성 
데이터가 여러 서버에 복제되기 때문에 
일부 데이터베이스 서버가 장애가 발생하더라도 데이터는 보존된다. 

또한 데이터베이스를 **여러 지역에 분산 저장**할 수도 있다.

#### 가용성 
하나의 데이터베이스 서버가 장애가 발생하더라도 
<br>다른 서버가 서비스를 계속 제공할 수 있다. 

즉 서비스 중단 가능성이 크게 줄어든다. 

### 장애 발생 시 동작 
#### Slave DB 장애 
- Slave DB가 하나뿐인 경우 
``` json
Slave Down
→ 모든 Read 요청을 Master DB가 처리
``` 
그리고 새로운 Slave DB가 추가된다. 

- Slave DB가 여러 개인 경우 
``` json
Slave1 Down
→ Read 요청을 Slave2 / Slave3가 처리
``` 

#### Master DB 장애 
Master DB가 다운되면 **Slave DB 중 하나가 새로운 Master로 승격(Promotion)** 된다. 
``` json
Master Down
Slave1 → New Master
``` 
이 과정을 **Failover**라고 한다. 
이후 새로운 Slave DB가 추가된다. 

### 확장된 시스템 구조 
로드 밸런서와 데이터베이스 다중화를 함께 사용하면 다음과 같은 구조가 된다. 
``` json
Client
   ↓
DNS
   ↓
Load Balancer
   ↓
Web Server 1
Web Server 2
   ↓
Master DB (Write)
   ↓
Slave DB 1 (Read)
Slave DB 2 (Read)
Slave DB 3 (Read)
``` 

동작흐름을 보면 
- 사용자는 DNS를 통해 로드 밸런서의 Public IP를 받는다. 
- 사용자는 해당 IP로 로드 밸런서에 접속한다. 
- 로드 밸런서는 요청을 여러 웹 서버로 분산한다. 
- 웹 서버는 조회 요청은 Slave DB에서 처리한다. 
- 데이터 변경 작업은 Master DB로 전달한다. 

## 캐시(Cache)
캐시란 **값비싼 연산 결과나 자주 참조되는 데이터를 메모리 안에 저장하여 이후 요청을 더 빠르게 처리할 수 있도록 하는 저장소**이다. 

웹 페이지를 새로고침 할 때마다 표시할 데이터를 가져오기 위해 **데이터베이스 호출**이 발생한다. 

애플리케이션의 성능은 **데이터베이스 호출 빈도**에 크게 영향을 받는다. 

이 문제를 완화하기 위해 **캐시(Cache)** 를 사용한다.

### 캐시 계층(Cache Tier)
캐시 계층(cache tier)은 **데이터가 잠시 보관되는 공간**으로, 데이터베이스보다 훨씬 빠르다. 

별도의 캐시 계층을 두면 
- 애플리케이션 성능 향상 
- 데이터베이스 부하 감소 
- 캐시 계층의 독립적 확장 

이 가능하다. 

### 캐시 서버 구조 

``` json
Application Server
        ↓
      Cache
        ↓
     Database
``` 

### 캐시 동작 방식 
1. 웹서버는 요청을 받으면 **캐시에 데이터가 있는지 확인**한다. 
2. 캐시에 데이터가 있다면 **해당 데이터를 바로 반환**한다. 
3. 캐시에 데이터가 없다면 **데이터베이스를 조회**한다. 
4. 조회한 데이터를 **캐시에 저장한 뒤 클라이언트에 반환**한다.

이 방식을 읽기 주도형 캐시 전략(Read-through caching strategy) 이라고 한다. 


### 캐시 사용시 유의할 점 
#### 1.캐시는 어떤 상황에 바람직한가
- 데이터 갱신(write)은 자주 발생하지 않는다. 
- 데이터 조회(read)는 빈번하게 발생한다.
즉, **읽기 요청이 많은 데이터에 캐시를 적용하는 것이 효과적이다.** 

#### 2.어떤 데이터를 캐시에 둘 것인가 
캐시는 **휘발성 메모리**에 데이터를 저장한다. 

따라서 다음 원칙을 지켜야한다. 
- 중요한 데이터는 **지속적 저장소(persistent data store)** 에 저장한다.
- 캐시는 **임시 데이터 저장소**로 사용한다. 

캐시 서버가 재시작되면 **캐시 데이터는 모두 사라질 수 있다.** 

#### 3.캐시에 보관된 데이터는 어떻게 만료되는가 (Expire)
캐시 데이터는 **만료 정책(expire policy)** 을 설정한다.

만료시간이 너무 짧으면 **데이터베이스 조회가 자주 발생한다.**

반대로 만료시간이 너무 길면 **원본 데이터와 캐시 뎅터가 달라질 수 있다.**

따라서 적절한 만료 정책을 설정하는 것이 중요하다. 

#### 4. 일관성(Consistency)은 어떻게 유지되는가 
일관성은 

**데이터 저장소의 원본 데이터와 캐시 데이터가 동일한 상태인지 여부**를 의마한다. 

다음과 같은 상황에서 일관성이 깨질 수 있다. 
``` json
DB 데이터 변경
↓
캐시 데이터는 이전 상태 유지
``` 
특히 분산 시스템 환경에서는 캐시와 저장소 사이의 일관성을 유지하는 것이 어려운 문제다. 

#### 5. 장애에는 어떻게 대처할 것인가 
캐시 서버가 하나만 존재하면 **단일 장애 지점(Single Point of Failure, SPOF)** 이 될 수있다.

SPOF란 

**특정 지점의 장애가 전체 시스템을 중단시키는 경우**를 의미한다. 

이를 해결하기 위해 
- 캐시 서버 다중화 
- 여러 지역에 캐시 서버 분산 

등의 방법을 사용한다. 

#### 6. 캐시 메모리는 얼마나 크게 잡을 것인가 
캐시 메모리가 너무 작으면 

데이터가 캐시에서 **자주 밀려나게(eviction)** 된다. 

이 경우 캐시 효율이 떨어진다. 

이를 방지하기 위해 
- 캐시 메모리를 충분히 확보한다. 
- 캐시 메모리를 과할당(overprovision) 하는 방법도 사용한다. 

#### 7. 데이터 방출 정책 (Eviction Policy)
캐시가 가득 차면 기존 데이터를 제거해야 한다. 

리를 **데이터 방출 정책(eviction policy)** 이라고 한다. 

대표적인 정책은 다음과 같다. 

##### LRU(Least Recently Used)
가장 오랫동안 사용되지 않은 데이터를 제거하는 방식 

가장 널리 사용되는 캐시 정책이다. 

##### LFU(Least Frequently Used)
가장 사용 빈도가 낮은 데이터를 제거하는 방식 

##### FIFO(First In First Out)
가장 먼저 캐시에 들어온 데이터를 먼저 제거하는 방식 

## 콘텐츠 전송 네트워크 (CDN)
**CDN(Content Delivery Network)** 이란 
<br> 정적 콘텐츠를 전송하기 위해 지리적으로 분산된 서버 네트워크이다. 

CDN은 다음과 같은 **정적 콘텐츠(static content)** 를 캐시하여 전달한다. 

- 이미지 
- 비디오 
- CSS
- JavaScript 

사용자와 가장 가까운 CDN 서버가 콘텐츠를 전달함으로써 **웹사이트 로딩 속도를 개선**할 수 있다. 

### CDN의 기본 동작 방식 
CDN은 **사용자와 가장 가까운 서버에서 콘텐츠를 제공**하는 방식으로 동작한다. 

사용자가 CDN 서버에서 멀수록 웹사이트 로딩 속도는 느려진다. 

예를 들어 
- CDN 서버가 샌프란시스코에 있을 경우 
- LA 사용자는 빠르게 로딩 
- 유럽 사용자는 상대적으로 느리게 로딩 

### CDN 동작 과정 
#### 1. 사용자 A가 이미지 URL로 접근한다. 

예시 
``` json
https://mysite.cloudfront.net/logo.jpg
https://mysite.akamai.com/image-manager/img/logo.jpg
``` 
이 도메인은 **CDN 서비스 사업자(CloudFront, Akamai등)** 가 제공한다. 

#### 2. CDN 서버 캐시에 이미지가 없는 경우 
CDN 서버는 **원본 서버(origin server)** 에 요청한다. 

원본 서버는 다음과 같은 곳일 수 있다. 
- 웹 서버 
- Amazon S3 같은 온라인 저장소 

#### 3. 원본 서버가 파일을 CDN 서버에 반환한다. 
HTTP 응답 헤더에는 TTL(Time-To-Live) 값이 포함된다. 
<br>TTL은 해당 콘텐츠가 얼마나 오래 캐시될 수 있는지를 의미한다.

#### 4. CDN 서버는 파일을 캐시하고 사용자 A에게 반환한다. 
이미지는 TTL 시간 동안 CDN에 저장 된다. 

#### 5. 사용자 B가 같은 이미지 요청을 보낸다. 

#### 6. CDN 캐시에 데이터가 존재하면 

원본 서버를 거치지 않고 캐시에 바로 응답한다. 

### CDN 구조 
``` json
User
  ↓
CDN Server
  ↓
Origin Server
(Web Server / Storage)
``` 

### CDN 사용시 고려해야 할 사항 
#### 1. 비용 
CDN은 일반적으로 **제3자 서비스 사업자(third-party provider)** 가 운영한다.


비용은 다음 기준으로 발생한다. 
- CDN으로 들어오는 데이터 
- CDN에서 나가는 데이터 

따라서 

자주 사용되지 않는 콘텐츠는 CDN에서 제외하는 것이 좋다. 

#### 2. 적절한 만료 시한 설정(TTL) 
TTL(Time-To-Live)은 **콘텐츠 캐시 유지** 시간을 의미한다. 

TTL 설정 시 고려 사항 

TTL이 너무 길면 -> 콘텐츠 시선도 감소 
<br>TTL 이 너무 짧으면 -> 원본 서버 접근 증가 

따라서 콘텐츠 특성에 맞는 TTL 설정이 중요하다. 

#### 3. CDN 장애 대응
CDN 장애가 발생할 경우 시스템 동작을 고려해야 한다. 

예를 들어 
- CDN이 응답하지 않으면 
- 클라이언트가 원본 서버(origin server)에서 직접 콘텐츠를 가져오도록 구성 

할 수 있다. 

#### 4. 콘텐츠 무효화(Invalidation) 
TTL이 아직 남아 있어도 CDN 캐시를 제거할 수 있다. 

방법은 다음과 같다. 

#### CDN API 사용 
CDN 서비스 제공자가 제공하는 API를 통해 캐시 삭제 

#### 오브젝트 버저닝 (Object Versioning)
콘텐츠의 버전을 변경하여 새로운 콘텐츠 제공 

예시 
``` json
image.png?v=2
``` 
URL 버전이 바뀌면 CDN은 새로운 콘텐츠로 인식한다. 

### CDN + Cache 구조 
시스템 확장구조에서는 다음과 같이 사용된다. 
 ``` json
Client
   ↓
CDN (정적 콘텐츠)
   ↓
Application Server
   ↓
Cache
   ↓
Database
``` 

### CDN 도입 후 변화 
#### 1. 정적 콘텐츠 처리 방식 변경 
기존
 ``` json
Client → Web Server → Static File
``` 
변경 후 
 ``` json
Client → CDN → Static File
``` 
정적 콘텐츠(JS, CSS, 이미지)는 **웹 서버가 아니라 CDN에서 제공**된다. 

#### 2. 데이터베이스 부하 감소 
- 캐시는 DB 조회 부하 감소 
- CDN은 정적 콘텐츠 부하 감소 

결과적으로 

전체 시스템이 성능이 개선된다. 

## 무상태(Stateless) 웹 계층
웹 애플리케이션에서 상태(State)는 **사용자 요청 사이에서 유지되어야 하는 데이터** 를 의미한다. 

대표적인 상태 정보는 다음과 같다. 
- 사용자 세션 데이터 
- 사용자 프로필 정보 
- 로그인 인증 정보 

이러한 상태 정보를 어디에 저장하느냐에 따라 **웹 아키텍처 구조가 달라진다.**

### 상태 정보 의존적인 아키텍처 (Stateful Architecture)
상태 정보 의존적인 아키텍처에서는 **웹 서버가 사용자 상태 정보를 직접 보관**한다. 

예를 들어 다음과 같은 구조가 될 수 있다. 

사용자 A -> 서버1 
사용자 B -> 서버2 
사용자 C -> 서버3 

각 서버는 자신에게 연결된 사용자의 상태 정보를  저장한다. 

 ``` json
[서버1]
사용자 A 세션 데이터
사용자 A 프로필 이미지

[서버2]
사용자 B 세션 데이터
사용자 B 프로필 이미지

[서버3]
사용자 C 세션 데이터
사용자 C 프로필 이미지
``` 

### 상태 저장 아키텍처의 문제점 
#### 1. 같은 서버로 요청이 전달되어야 한다. 

사용자 A의 세션 정보가 서버1에 저장되어 있다면 
<br> 사용자 A의 모든 요청은 항상 서버1로 전달되어야 한다. 

만약 요청이 다른 서버로 전달되면 다음과 같은 문제가 발생한다. 

``` json
사용자 A -> 서버2 요청 
->서버2에는 사용자A세션 데이터 없음 
-> 인증 실패 
``` 

#### 2. Sticky Session 필요 
로드밸런서는 같은 사용자의 요청을 항상 같은 서버로 보내야 한다. 

이를 **고정 세션(Sticky Session)** 이라고 한다.

Sticky Session은 로드밸런서가 특정 사용자를 특정 서버에 고정시키는 기능이다. 

#### 3. 서버 확장이 어려워진다.
웹 서버를 추가하거나 제거할 때 문제가 발생한다. 

예를 들어 서버가 하나 추가 되면 

``` json
서버1
서버2 
서버3
서버4 
``` 
새 서버에는 기존 사용자 세션 데이터가 없기 때문에 로드밸런싱이 복잡해진다. 

#### 4. 장애 대응이 어렵다 
예를 들어 서버1이 장애로 다운되면 

서버1에 저장되어 있던 사용자 세션 정보가 모두 사라진다. 

이 경우 해당 사용자들은 **다시 로그인해야 하는 문제**가 발생한다. 

### 무상태(Stateless) 아키텍처 
무상태 아키텍처에서는 **웹 서버가 상태 정보를 저장하지 않는다.** 

상태 정보는 웹 서버가 아닌 **공유 저장소(shared storage)** 에 저장된다.

예시 구조
``` json
사용자
↓
로드밸런서
↓
웹 서버들
↓
공유 저장소
``` 
웹 서버는 상태 정보가 필요할 경우 **공유 저장소에서 데이터**를 가져온다. 

### 무상태 아키텍처 특징 
이 구조에서는 사용자 요청이 **어떤 웹 서버로 전달되어도 문제가 없다.** 

웹 서버는 상태 정보를 보관하지 않기 때문이다. 

필요한 데이터는 공유 저장소에서 조회한다. 

### 상태 정보 저장 위치 
상태 정보는 다음과 같은 저장소에 저장할 수 있다. 
- 관계형 데이터베이스(RDB) 
- 캐시 시스템(Redis, Memcached)
- NoSQL 데이터베이스 

규모 확장이 쉽기 때문에 NoSQL을 사용하는 경우가 많다. 

### 무상태 웹 계층의 장점 
#### 1. 단순한 구조 
웹 서버가 상태 정보를 저장하지 않기 때문에 
<br> 아키텍처가 단순해진다. 

#### 2. 안정성 향상 
특정 서버가 장애가 발생하더라도 
<br> 다른 서버가 요청을 처리할 수 있다. 

#### 3. 수평 확장 용이 
트래픽이 증가하면 **웹 서버를 추가** 하면 된다. 

트랙피이 감소하면 **웹 서버를 제거** 할 수 있다. 

이를 **자동 규모 확장(Auto Scaling)** 이라고 한다. 

## 데이터 센터 (Data Center)

### 1. 데이터 센터란 
데이터 센터는 **서비스를 제공하는 물리적 서버 집합**이다. 

대규모 시스템에서는 **여러 지역에 데이터 센터를 분산 배치**하여 

- 가용성(Availability)
- 성능(Performance)

을 높인다. 

## 2. 지리적 라우팅 (Geo Routing)
여러 데이터 센터를 사용할 경우 
사용자는 가장 가까운 데이터 센터로 연결된다. 

이 과정을 **지리적 라우팅(Geo Routing)** 이라고 한다. 


## GeoDNS 

GeoDNS 는 **사용자의 위치에 따라 도메인을 다른 IP주소로 변환하는 DNS 서비스**이다. 

예시 
``` json
x% 사용자 → US-East 데이터 센터

(100 - x)% 사용자 → US-West 데이터 센터
``` 

## 3. 정상 상황에서의 트래픽 흐름
장애가 없는 경우 
- 사용자는 가장 가까운 데이터 센터로 연결된다. 
- 트래픽은 여러 데이터 센터로 분산된다. 

이를 통해 
- 응답 속도 개선 
- 부하 분산 

이 가능하다.

## 4. 장애 발생 시(Failover)
특정 데이터 센터에 장애가 발생하면 

모든 트래픽은 **정상 데이터 센터로 자동 전환**된다. 

예시 
``` json
US-West 장애 발생
→ 모든 트래픽 → US-East로 전달
``` 

이러한 동작을 통해 
- 서비스 중단 없이 운영 가능 
- 고가용성 확보 

가 가능하다. 

## 5. 다중 데이터 센터 구성 시 고려 사항 

### 1. 트래픽 우회(Traffic Routing) 
사용자를 올바른 데이터 센터로 전략이 필요하다. 

대표적인 방법으로는 GeoDNS 가 있다 

### 2. 데이터 동기화 (Data Synchronization)
각 데이터 센터가 별도의 데이터베이스를 사용하는 경우 문제가 발생할 수 있다. 

문제 상황 
``` json
DB1 (US-East)
DB2 (US-West)

US-West 장애 발생
→ 트래픽이 US-East로 이동

하지만

→ 필요한 데이터가 없을 수 있음
``` 

### 해결방법 
데이터를 여러 데이터 센터에 걸쳐 **다중화(Replication)** 한다.
- 동일한 데이터를 여러 센터에 복제 
- 장애 시에도 데이터 일관성 유지 

### 3. 테스트와 배포 (Testing & Deployment)
여러 데이터 센터를 사용하는 경우 

다음이 중요하다. 

#### 테스트 
- 다양한 지역에서 서비스 동작 확인 
- 지역별 네트워크 지역 확인 

#### 배포 
- 모든 데이터 센터에 동일한 서비스 배포 필요 
- 자동화된 배포 시스템 필요 

## 6. 메시지 큐 (Message Queue)
### 1.메세지 큐란 
메시지 큐(Message Queue)는

**비동기 통신(asynchronous communication)을 지원하는 컴포넌트**로, 메세지를 안전하게 저장하고 전달하는 역할을 한다. 

### 핵심 특징 
- 메세지 무손실 보장 (Durability) 
<br>-> 큐에 저장된 메세지는 **소비자가 처리할 때까지 안전하게 보관**된다. 
-  메세지 버퍼 역할 
<br>-> 생산자와 소비자 사이에서 **중간 저장소 역할 수행** 
- 비동기 처리 지원 
<br>-> 요청을 즉시 처리하지 않고 **나중에 처리 가능**

### 2. 메세지 큐 기본 구조 
메세지 큐의 기본 구성 요소는 다음과 같다. 
- 생산자 (Producer / Publisher)
- 메시지 큐 (Queue)
- 소비자 (Consumer / Subscriber)

### 동작 흐름
1. 생산자가 메세지를 생성한다. 
2. 메세지를 큐에 발행(publish)한다. 
3. 메세지는 큐에 저장된다. 
4. 소비자가 메세지를 가져간다. 
5. 소비자가 해당 작업을 처리한다. 

### 3. 메세지 큐의 장점 

#### 1) 느슨한 결합 (Loose Coupling)
메세지 큐를 사용하면 
- 생산자와 소비자가 직접 연결되지 않는다. 
- 서로 독립적으로 동작할 수 있다. 

#### 2) 독립적인 확장성 
- 생산자와 소비자는 각각 따로 확장할 수 있다. 
- 시스템 전체 확장이 쉬워진다. 

#### 3) 장애 내성 (Fault Tolerance)
다음과 같은 상황에서도 시스템이 동작한다. 
- 소비자가 다운되어도 -> 메세지는 큐에 저장됨 -> 나중에 처리 가능 
- 생산자가 일시적으로 중단되어도 -> 소비자는 큐에서 계속 처리 가능 

### 4. 메세지 큐 사용 예시 

#### 이미지 처리 시스템 
사진 보정 기능을 예로 들 수 있다. 

예시 작업 
- 이미지 크롭(Cropping)
- 샤프닝 (Sharpening)
- 블러링 (Blurring)

#### 처리 흐름 
1. 웹 서버가 이미지 처리 요청을 받는다. 
2. 해당 작업을 메세지 큐에 넣는다 
3. 작업 프로세스(Worker)가 큐에서 작업을 가져온다. 
4. 비동기적으로 작업을 수행한다. 

### 5. 확장 전략 
메세지 큐를 사용하면 

생산자와 소비자의 규모를 **독립적으로 조절할 수 있다.** 

#### 1) 큐가 쌓이는 경우
- 작업이 많음 
- 큐에 메세지가 계속 쌓임 

-> 해결 방법 
- Worker(소비자)를 늘린다. 

#### 2) 큐가 비어 있는 경우 
- 작업이 적음 

-> 해결방법 
- Worker 수를 줄인다. 

### 6. 정리 
메세지 큐는 
- 비동기 처리 
- 느슨한 결합 
- 안정적인 메세지 전달 

을 가능하게 하는 핵심 컴포넌트이다. 

특히 대규모 시스템에서는 
- 서비스 간 의존성을 줄이고 
- 독립적인 확장을 가능하게 하기 때문에 

**확장성과 안정성을 동시에 확보할 수 있다.**
