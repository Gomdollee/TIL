# 동시성 문제와 Race Condition

## 1. 동시성 문제의 본질 
멀티스레드 환경에서 문제가 발생하는 핵심 이유는 **여러 스레드가 동일한 데이터(공유 자원)을 동시에 다루기 때문**이다.

이때 중요한 개념이 **임계 영역(Critical Section)** 이다.
 - 공유 자원을 읽거나 수정하는 코드 구간 
 - 동시에 접근하면 데이터가 깨질 수 있는 영역 

## 2. 경쟁상태(Race Condition)
### 정의 
여러 스레드가 동일한 임계 영역을 동시에 실행하면서 **실행 순서에 따라 결과가 달라지는 현상** 

### 왜 위험할까? 
- 실행 결과가 매번 다름(비결정성)
- 테스트에서는 안 터지고 운영에서 터짐 
- 디버깅 거의 불가능 수준 

결국 데이터 정합성 깨짐 = 서비스 장애로 직결 

## 3. Race Condition이 발생하는 이유 
Race Condition은 단순히 "동시에 접근해서" 가 아니라 아래와 같은 2가지 이유때문에 발생한다. 

### 3.1 원자성 (Automicity) 파괴 
개발자는 `i++`를 하나의 연산으로 생각하지만 실제 CPU 레벨에서는 여러 단계로 나뉜다. 
- 값 읽기(Read)
- 값 변경(Modify)
- 값 저장(Write)

이 과정 사이에 다른 스레드가 끼어들 수 있다. 

결과 
- 값이 덮어씌워짐 (Lost Update)
- 기대값보다 작은 결과 발생 

### 3.2 가시성 (Visibility) 문제
멀티코어 환경에서는 각 CPU가 **자기 캐시**를 사용한다. 

문제는 한 스레드가 값을 변경해도 다른 스레드가 그걸 **즉시 못 볼 수 있다.**

결과 
- 최신 데이터가 공유되지 않음 
- 오래된 값을 기준으로 로직 수행 

### 3.3 순서 보장(Ordering)
컴파일러/CPU는 성능을 위해 코드 실행 순서를 바꿀 수 있다. 

- 단일 스레드에서는 문제 없음 
- 멀티스레드에서는 치명적임 

## 4. 결국 무엇이 보장되어야 하나? 
### 1. 원자성 
- 연산이 쪼개지지 않고 한 번에 실행 
### 2. 가시성 
- 한 스레드의 변경이 다른 스레드에 즉시 반영 
### 3. 순서 보장 
- 명령 실행 순서가 의도대로 유지 

## 5. 해결 전략(Java 기준)
### 5.1 Lock 기반(Blocking)

synchronized / Lock 
- 한 번에 하나의 스레드만 접근 
- 내부적으로 메모리 동기화까지 수행 

특징 
- 가장 안전
- 대신 성능 비용 존재 (대기, context switcing)

### 5.2 Lock-Free (Non-blocking)

Atiomic 클래스(CAS 기반)
- 값 비교 후 교체 (Compare-And-Swap)
- 실패 시 재시도 

특징 
- 락 없음 -> 빠름 
- 충돌 많으면 성능 저하(spin)

### 5.3 volatile 
- CPU 캐시를 거치지 않고 메인 메모리 사용 
- 값 변경이 즉시 다른 스레드에 보임
