# 2장. 개략적인 규모 추정
시스템 설계 면접에서는 **시스템 용량이나 성능을 대략적으로 계산**해보라는 질문이 자주 나온다. 

이를 **개략적 규모 추정(Back-of-the-envelope estimation)** 이라고 한다. 

### 개략적 규모 추정이란 
**보편적인 성능 수치를 기반으로 사고 실험을 통해 시스템 규모를 추정하는 것**이다. 

목적은 
- 설계가 요구사항을 만족하는 판단 
- 시스템이 감당 가능한지 검증 

이고 이를통해 
- 시스템 병목 예상 
- 인프라 규모 판단 
- 설계 방향 결정 

이 가능하다. 

## 1. 2의 제곱수 
데이터 크기를 계산할 때는 

**10진수(1000)가 아니라 2의 제곱 기준으로 생각** 해야한다. 

## 📊 데이터 단위 (2의 제곱 기준)

| 2의 제곱 | 근사값 | 단위 | 의미 |
|---|---|---|---|
| 2¹⁰ | 1천 | KB | Kilobyte |
| 2²⁰ | 1백만 | MB | Megabyte |
| 2³⁰ | 10억 | GB | Gigabyte |
| 2⁴⁰ | 1조 | TB | Terabyte |
| 2⁵⁰ | 1000조 | PB | Petabyte |

### 기본 단위 
- 1Byte = 8bit 
- ASCII 문자 1개 = 1 Byte

## 2. 모든 프로그래머가 알아야 하는 응답 지연 

구글의 jeff Dean은 
컴퓨터 시스템에서 발생하는 다양한 연산들의 **응답 지연(latency)** 값을 정리하였다. 

이 값들은 시간이 지나며 일부 변경되었지만 
**각 연산의 상대적인 속도 차이를 이해하는 기준**으로 여전히 중요하다. 

### 1. 주요 응답 지연 값

📊 Latency 기준표
| 연산 | 시간 |
|---|---|
| L1 캐시 참조 | 0.5ns |
| 분기 예측 실패 | 5ns |
| L2 캐시 참조 | 7ns |
| Mutex lock/unlock | 100ns |
| 메모리 접근 | 100ns |
| 1KB 압축 | 10μs |
| 2KB 네트워크 전송 | 20μs |
| 메모리에서 1MB 읽기 | 250μs |
| 같은 데이터센터 내 RTT | 500μs |
| 디스크 탐색 (seek) | 10ms |
| 디스크에서 1MB 읽기 | 30ms |
| 대륙 간 RTT | 150ms |

### 2. 시간 단위 정리 
- 1 ns = 10⁻⁹ 초
- 1 μs = 10⁻⁶ 초 = 1,000 ns
- 1 ms = 10⁻³ 초 = 1,000 μs = 1,000,000 ns 

### 3. 핵심 인사이트 
#### 1) 메모리는 빠르고 디스크는 매우 느리다 
- 메모리 접근: 약 100ns 
- 디스크 접근: ms 단위 

#### 2) 디스크 탐색은 피해야 한다. 
- seak 자체가 매우 비싼 연산 
- 랜덤 I/O 보단 순차 I/O가 훨씬 유리

#### 3) 압축은 생각보다 빠르다 
- 네트워크보다 압축이 더 빠를수 있음 
- 전송 전에 압축하는 것이 유리 

#### 4) 네트워크 비용은 크다 
- 같은 DC: μs
- 대륙 간: ms

#### 5) 데이터 센터 간 통신은 느리다 
- region 간 통신은 높은 latency 발생
