# 1장. 사용자 수에 따른 규모 확장성 
대규모 시스템 설계는 한 번에 완성되는 것이 아니라 **지속적인 확장과 개선의 과정** 



## 1.  단일서버 
웹, 앱, 데이터베이스,캐 시 등이 전부 서버 한대에서 실행된다고 하자 

사용자 요청의 처리 흐름을 살펴보면 

### DNS 조회 
- 사용자는 보통 IP 주소가 아닌 도메인 이름을 통해 서비스에 접속한다. ex) www.naoor.com
- 하지만 인터넷 통신은 실제로 IP 주소를 기반으로 이루어지기 때문에 도메인을 IP주소로 변환하는 과정이 필요하다. 
- 이 역할을 하는게 **DNS(Domain Name System)** 이다. 
- DNS 서버는 일반적으로 외부(유료) 서비스를 사용한다. (Google DNS, AWS Route53 ...)

### HTTP 요청 전달 
- DNS 조회를 통해 얻은 IP주소로 HTTP 요청이 전달된다. 
- ex) GET /users/123 
- HTTP 요청에는 다음 정보들이 표함된다. 
  - Method(GET/POST/PUT/DELETE)
  - URL 
  - Header 와 Body
  
### 서버 응답 반환 
- 서버는 요청 처리 후 클라이언트에게 **응답** 을 반환한다.
- 응답 형식은 보통 두가지로 구성된다.  
  - **HTML 응답** 
    - 웹 브라우저가 직접 렌더링할 때 사용된다. 
    - Content-Type: text/html
  - **JSON 응답**
    - API 기반 서비스에서 가장 많이 사용하는 형태이다. 
    - Content-Type: application/json
``` json
  {
  "id": 12,
  "firstName": "John",
  "lastName": "Smith",
  "address": {
    "streetAddress": "21 2nd Street",
    "city": "New York",
    "state": "NY",
    "postalCode": 10021
  },
  "phoneNumbers": [
    "212-555-1234",
    "646-555-4567"
  ]
}
```

### 클라이언트 요청 
클라이언트 요청은 두가지 단말로부터 오는데 하나는 웹 앱이고 하나는 모바일 앱이다. 
- **웹 앱** 
  - 구성요소는 서버(Java, Python)언어와 클라이언트 구현용 언어(HTML, 자바스크립트) 가있다.
  - Browser -> HTTP 요청 -> Web Server -> Database 구조로 동작한다. 
- **모바일 앱** 
  - 모바일 앱 역시 서버와 통신하기 위해 HTTP API를 사용한다. 
  - 모바일 앱은 화면은 앱 내부에서 처리하고 데이터만 JSON API로 받아온다. 



## 2.  데이터베이스
사용자가 증가하면서 하나의 서버에서 모든 기능을 처리하기 어려워진다. 
따라서 **역할별로 분리**하는 구조이다. 

대표적 방식이 웹 계층(Web Tier)과 데이터 계층(Data Tier) 분리이다. 

### 데이터베이스 종류 
데이터 저장소는 크게 두 가지 유형으로 나뉜다. 
 - 관계형 데이터베이스 (RDBMS)
 - 비관계형 데이터베이스 (NoSQL) 

### 관계형 데이터베이스 (RDBMS) 
관계형 데이터베이스는 **테이블 기반 구조로 데이터를 저장**하는 데이터베이스이다. 

- MySQL, 오라클 데이터베이스, PostgreSQL 등 
- 데이터는 테이블 형태로 저장되며 SQL을 통해 조회 및 조작한다. 

특징 
- 데이터 구조가 명확 
- 데이터 무결성 보장 
- 트랜잭션 지원 
- JOIN 연산 지원 

### 비관계형 데이터베이스 (NoSQL) 
비관계형 데이터베이스는 **테이블 기반 구조가 아닌 다양한 형태로 데이터를 저장**하는 데이터베이스이다. 
- CouchDB, Neo4j, Cassandra, HBase, Amazon DynamoDB 등
- 대부분의 NoSQL 데이터베이스는 JOIN 연산을 지원하지 않는다. 
- 대신 데이터를 비정규화 형태로 저장하는 경우가 많다. 

### NoSQL 데이터베이스 종류 
NoSQL은 크게 네 가지 유형으로 나뉜다. 

#### KEY-VALUE Store (키-값 저장소)
키와 값 형태로 데이터를 저장하는 방식 
``` json
key : user:12
value : {name:"John", age:30}
``` 
대표 시스템 
- Redis
- DynamoDB

특징 
- 조회 속도가 매우 빠름 
- 캐시 시스템으로 많이 사용됨 

#### Graph Store (그래프 저장소)
데이터 간 관계 구조를 그래프로 표현하는 데이터베이스 

대표 시스템
- Neo4j

특징
- 관계 중심 데이터 분석에 유리 
- 추천 시스템 등에 사용  

#### Column Store (칼럼 저장소)
데이터를 열 기반 구조로 저장하는 데이터베이스 

대표 시스템 
- Cassandra 
- HBase 

특징 
- 대용량 데이터 처리에 강함 
- 분산 시스템에 적합 

#### Document Store (문서저장소)
JSON 형태 문서를 저장하는 데이터베이스 
``` json
{
  "id": 12,
  "name": "John",
  "age": 30
}
``` 
대표 시스템 
- MongoDB 
- CouchDB 

특징 
- 스키마가 유연함 
- JSON 데이터 저장에 적합 

### NoSQL을 사용하는 경우 

#### 매우 낮은 응답 지연이 필요한 경우 
- 캐시 시스템, 실시간 서비스 
#### 비정형 데이터 처리 
- 데이터 구조가 고정되지 않은 경우 (로그데이터, 사용자 행동 데이터, JSON 데이터)
#### 대용량 데이터 처리 
- 빅테이터 시스템, 로그 분석 시스템 
#### 직렬화 데이터 저장 
- JSON, YAML, XML 등 문서 기반 데이터 저장이 필요한 경우 

## 3. 수직적 규모 확장 vs 수평적 규모 확장 
서비스의 사용자 수와 트래픽이 증가하면서 시스템의 처리 능력을 확장해야 한다. 
이때 시스템을 확장하는 방식은 크게 두 가지로 나뉜다. 
- 수직적 규모 확장(Vertical Scaling)
- 수평적 규모 확장(Horizontal Scaling)
  
이 두 방식은 시스템 설계에서 **성능 확장 전략의 기본 개념** 이다.

### 수직적 규모 확장(Vertical Scaling)
수직적 규모 확장은 **하나의 서버 성능을 높이는 방식**이다.
일반적으로 **Scale Up** 이라고 부른다. 

예를 들어 기존 서버의 자원을 업그레이드 하는 방식이다. 
- CPU 성능 증가 
- RAM 용량 증가 
- 디스크 성능 향상(SSD)
- 네트워크 대역폭 증가 

이 방식의 가장 큰 장점은 **구조가 단순하다는 점**이다. 
<br>서버를 추가하거나 시스템 구조를 변경할 필요 없이 **하드웨어만 업그레이드** 하면 된다. 
<br>그래서 초기 서비스나 트래픽이 적은 시스템에서 **수직적 확장이 가장 간단한 선택**이 된다. 

#### 수직적 확장의 한계 
하지만 수직적 확장에는 몇 가지 중요한 한계가 존재한다. 

#### 물리적인 확장 한계 
서버의 CPU와 메모리를 **무한히 늘릴 수는 없다**.
<br>결국 하나의 서버가 처리할 수 있는 성능에는 **물리적인 제한**이 존재한다. 

#### 장애 대응이 어렵다
수직적 확장은 **서버가 하나인 구조**를 유지하기 때문에 
<br>서버 장애가 발생하면 서비스 전체가 중단된다. 

#### 대규모 트래픽 대응이 어렵다
사용자가 매우 많아지면 하나의 서버로는 요청을 처리하기 어렵다. 
<br> 예를 들어 하나의 서버에 사용자 10만명이 몰리면 서버가 과부하 상태가 되며 응답속도지연, 요청실패, 서버 다운같은 문제가 발생할 수있다. 

### 수평적 규모 확장(Horizontal Scaling)
수평적 규모 확장은 **서버를 여러 대 추가하여 시스템 성능을 확장하는 방식**이다. 
일반적으로 **Scale Out** 이라고 부른다. 

- Web Server 1대 -> Web Server 5대 
이 방식은 여러 서버가 동시에 요청을 처리하기 때문에 **대규모 트래픽 처리에 유리**하다. 

#### 수평 확장의 핵심 요소 
수평 확장 구조에서는 **로드 밸런서(Load Balancer)** 가 중요한 역할을 한다. 
<br> 로드 밸런서는 **여러 서버로 트래픽을 분산시키는 장치** 이다.

로드 밸런서는 사용자 요청을 **여러 서버에 균항하게 분배**하여 특정 서버에 부하가 집중되는 것을 방지한다. 

#### 수평 확장의 장점 
수평 확장은 대규모 서비스에서 매우 중요한 확장 전략이다. 

#### 높은 확장성 
트래픽이 중가하면 **서버를 계속 추가할 수 있다.** 

#### 장애 대응 가능 
여러 서버가 존재하기 때문에 **한 서버가 장애가 발생해도 서비스는 계속 운영**될 수 있다. 

#### 트래픽 분산 
로드 밸런서를 통해 요청을 분산하여 **서버 부하를 줄일 수 있다.**
