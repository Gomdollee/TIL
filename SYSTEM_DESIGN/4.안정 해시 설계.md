# 안정 해시 설계
수평적 규모 확장성을 달성하기 위해서는 요청 또는 데이터를 서버에 균등하게 나눈것이 중요. 
안정 해시는 이 목표를 달성하기 위해 보편적으로 사용하는 기술이다. 

## 해시 키 재배치(rehash)문제 
N개의 캐시 서버가 있을 때,
데이터를 균등하게 분산하기 위해 일반적으로 아래와 같은 방식을 사용한다. 

 ``` java
serverIndex = hash(key) % N
``` 
이 방식은 각 key에 대해 해시 값을 계산한 뒤 
서버 개수(N)로 나눈 나머지를 이용해 저장할 서버를 결정한다. 

### 여기서 서버가 4대라고 가정하면(N=4)
 ``` java
serverIndex = hash(key) % 4
``` 
각 key는 다음과 같이 서버에 분산된다. 
- server 0 → key1, key3
- server 1 → key0, key4
- server 2 → key2, key6
- server 3 → key5, key7

-> 해시 값을 기준으로 비교적 고르게 분산됨 

이 방식은 
- 서버 수가 고정되어 있고
- 데이터 분포가 균등할 때는 잘 동작한다. 

### 이제 server 1이 장애로 제거되었다고 가정한다. 
-> 서버 수: 4- -> 3 
``` java
serverIndex = hash(key) % 3
``` 
같은 key들을 다시 계산하면 다음과 같이 변경된다. 
- server 0 → key0, key1, key5, key7
- server 1 → key2, key4, key6
- server 2 → key3

### 문제 상황 분석 
중요한 점은, 단순히 장애가 난 server 1의 데이터만 이동하는 것이 아니라 거의 모든 key의 위치가 변경된다. 

즉,
- 기존에 server2에 있던 key도 다른 서버로 이동
- 기존에 server3에 있던 key도 이동 

-> 전체 데이터가 다시 섞이는 구조 

### 문제의 결과 
이로 인해 다음과 같은 문제가 발생한다. 
### 1. 캐시 미스 증가 
클라이언트는 이전 위치를 기준으로 요청하지만 
데이터는 이미 다른 서버로 이동한 상태 

- 존재하지 않는 서버에서 조회
- Cache Miss 발생 

### 2. 시스템 부하 증가 
- 데이터가 대량으로 재배치됨
- 캐시가 무력화됨
- DB 또는 백엔드 시스템에 부하 증가 

### 결론 
`hash(key) % N` 방식은 서버 수가 변하면 대부분 데이터가 재배치 되는 문제가 있다. 
- 이로 인해 대규모 캐시 미스가 발생하며 
- 시스템 안정성이 크게 떠렁진다.

따라서 이 문제를 해결하기 위해 **안정 캐시(Consistent Hashing)** 가 상요된다. 

## 안정 해시 (Consistent Hashing)
### 안정 해시
>안정 해시는 해시 테이블 크기가 변경될 때 평균적으로 **k/n개의 키만 재배치 되는 해시 기술** 이다. 

- k: 전체 키 개수 
- n: 슬롯(또는 서버)개수 

기존 방식 
- 거의 모든 키 재배치 
안정 해시 
- 일부 키만 이동 

### 해시 공간과 해시 링 
- 해시 함수 : SHA-1 사용(예시)
- 해시 값 범위 : 0 ~ 2¹⁶⁰ - 1

이 범위를 직선으로 보면 "해시 공간" 

이 공간을 둥글게 이어 붙이면 **해시 링(Hash Ring)

### 해시 서버 
- 서버도 해시 함수로 변환 
- -> 해시 링 위 특정 위치에 배치 

예시 
- server0 -> 링의 한 지점 
- server1 -> 또 다른 지점 
- ...

중요한 점은 기존 방식처럼 **% 연산을 사용하지 않음** 

### 해시 키 
- key도 동일하게 해시 함수 적용 
- -> 링 위 특정 위치에 배치 

### 서버 조회 
어떤 key 가 저장될 서버는 : 

-> **해당 key 위치에서 시계 방향으로 이동하면서 처음 만나는 서버** 

예 : 
- key0 -> server0 
- key1 -> server1
- key2 -> server2 
- key3 -> server3 

### 서버 추가 
새로운 서버가 추가되면 : 

-> 전체가 아니라 **일부 key만 이동**

예 : 
- 기존 : key0 -> server0 
- 서버4 추가 후 : key0 -> server4 

이유 : 
- key0 위치에서 시계 방향으로 탐색 시 server4가 먼저 나오기 때문 
- 나머지 key는 그대로 유지 

### 서버 제거 
서버가 제거 되면 : 해당 서버가 담당하던 key만 이동 

예 : 
- server1 제거 
- key1 -> server2로 이동 

-> 나머지 key는 영향 없음 

### 기본 구현법의 두 가지 문제 
### 1. 파티션 불균형 문제 
- 파티션 = 서버 사이 해시 공간 

문제 : 서버 간 간격이 균등하지 않음 

결과 : 
- 어떤 서버 -> 매우 큰 영역 담당 
- 어떤 서버 -> 매우 작은 영역 담당 

### 2. 키 분포 불균형 문제 
문제 : key가 특정 서버에 몰릴 수 있음 

예 : 
- server2에 대부분 key 집중 
- server1, server3은 거의 없음 

### 가상 노드 (Virtual Node)
이 문제를 해결하기 위한 기법 
개념 
- 하나의 서버를 여러 개의 노드로 쪼개서 배치 

예 
- server0 → s0_0, s0_1, s0_2
- server1 → s1_0, s1_1, s1_2

하나의 서버가 여러 위치를 가지게 됨 

### 동작 방식 
- key -> 시계 방향 탐색 
- -> 가장 먼저 만나는 **가상 노드** 
- -> 해당 가상 노드가 속한 **실제 서버로 저장**

### 효과 
데이터 분포가 훨씬 균등해짐 
- 가상 노드 수 ↑ → 균등성 ↑
- 표준 편차 감소 

참고 
- 100~200개 가상 노드 사용 시 -> 편차 약 5~10% 

Trade-off
|항목 | 설명 |
|---|---|
|장점 | 데이터 분산 균등 |
|단점 |	메모리 사용 증가 |

-> 적절한 개수 선택 필요

### 재배치할 키 결정 
### 1. 서버 추가 
새 서버(s4)가 추가되면 영향 범위는 **s4 ~(반시계 방향 첫 서버s3) 사이**

-> 이 구간의 key 만 이동 

### 1. 서버 제거 
서버(s1)가 제거되면 영향 범위는 **s1 ~ (반시계 방향 첫 서버 s0) 사이**

-> 해당 key들은 다음 서버(s2)로 이동


| 구분    | 기존 해시    | 안정 해시     |
| ----- | -------- | --------- |
| 방식    | hash % N | hash ring |
| 서버 변화 | 전체 재배치   | 일부만 이동    |
| 확장성   | 낮음       | 높음        |
