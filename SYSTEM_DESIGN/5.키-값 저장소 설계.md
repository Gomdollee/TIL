# 키-값 저장소 설계
### 키-값 저장소 
**비관계형(non-relational) 데이트베이스**의 한 종류이다. 
- 데이터는 **key -> value 형태**로 저장 된다. 
- 이 관계를 키-값 쌍(pair) 이라고 한다. 

### 특징 
- key는 **유일해야 한다**
- value **key를 통해서만 접근 가능**
- value 타입은 제한 없음 

즉, 단순하지만 유연한 구조 

### 키 (Key)
- 일반 텍스트 가능 -> `last_logged_in_at`
- 해시 값 가능 -> `253DDEC4`

-> 성능상 **키는 짧을수록 좋다**

### 값 (Value)
값은 다양한 형태 가능 
- 문자열 (String)
- 리스트 (List)
- 객체 (Object)

-> 시스템 value의 구조를 신경 쓰지 않는다

### 대표적인 시스템 
- Amazon DynamoDB
- Memcached
- Redis

### 기본 연산 
키-값 저장소는 일반적으로 두 가지 연산을 지원한다.
 ``` java
put(key, value)
``` 
-> 데이터를 저장 
 ``` java
get(key)
``` 
-> key로 value 조회

## 문제 이해 및 설계 범위 확정
완벽한 설계는 없으며, 다음 요소들 사이에서 **균형(trade-off)** 을 잡는 것이 중요하다.

### 고려해야 할 요소 
- 읽기 vs 쓰기 성능
- 메모리 사용량 
- 일관성(Consistency) vs 가용성(Availability)

### 이번 장 설계 목표 
- 키-값 크기: 500KB 이하
- 대용량 데이터 저장 가능
- 높은 가용성 -> 장애 상황에서도 빠르게 응답
- 높은 확장성 -> 트래픽 증가 시 자동 확장
- 일관성 수준 조정 가능
- 낮은 지연시간 (Low Latency)

## 단일 서버 키-값 저장소 
가장 단순한 설계는 다음과 같다. 

-> **메모리 기반 해시 테이블** 

### 장점 
- 매우 빠른 속도 
- 구현이 단순 

### 문제점 
모든 데이터를 메모리에 저장해야 함 
- 데이터 커지면 불가능 
- 메모리 한계 존재 

### 개선 방법 
### 1. 데이터 압축 (Compression)
- 저장 공간 절약 

### 2. 메모리 + 디스크 분리 
- 자주 사용하는 데이터 -> 메모리 
- 나머지 데이터 -> 디스크 

### 한계 
이러한 개선에도 불구하고 

-> 결국 단일 서버로는 한계 도달 
- 데이터 증가
- 트래픽 증가 

### 결론 
대규모 시스템이서는 **분산 키-값 저장소 (Distributed KV Store)** 필요

## 분산 키-값 저장소 
- 분산 키-값 저장소 = **분산 해시 테이블 (DHT)**
- 데이터를 여러 서버에 분산 저장 

**확장성과 가용성 확보**

### CAP 정리 
분산 시스템에서는 다음 3가지를 동시에 만족할 수 없다. 
- Consistency(일관성)
  - 모든 클라이언트가 동일한 데이터 조회 
- Availability(가용성)
  - 일부 노드 장애에도 항상 응답 
- Partition Tolerance(파티션 감내)
  - 네트워크 장애 발생해도 시스템 유지 

### 시스템 유형 
- CP : 일관성 + 파티션 감내(가용성 희생)
- AP : 가용성 + 파티션 감내(일관성 희생)
- CA : 일관성 + 가용성 (현실적으로 불가능)

현실 시스템 -> 반드시 partition 고려 
따라서 CP 또는 AP 선택 

### 분산 시스템의 현실 
- 네트워크 장애는 필연적 
- 데이터는 여러 노드에 복제됨 

문제상황 
- 일부 노드가 끊기면 데이터 불일치 발생 

### 선택 
- CP 시스템 
  - 일관성 유지 위해 쓰기 중단 
  - (은행 시스템)
- AP 시스템 
  - 가용성 유지 
  - 나중에 데이터 동기화
