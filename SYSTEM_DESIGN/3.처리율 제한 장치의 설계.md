# 처리율 제한 장치의 설계 
## 1.처리율 제한 장치란 
네트워크 시스템에서 처리율 제한 장치(rate limiter)는 
<br> 클라이언트 또는 서비스가 보내는 트래픽의 처리율(rate)을 제어하기 위한 장치다. 

HTTP 기준으로 보면 
<br> 일정 기간 동안 전송되는 요청 횟수를 제한한ㄷ. 

- 정의된 임계치(threshold)를 초과하면 
-> 이후 요청은 처리되지 않고 차단(block)된다. 

### 사용 사례 
- 사용자는 초당 2회 이상 글을 올릴 수 없다 
- 같은 IP 주소로 하루 10개 이상 계정 생성 불가 
- 같은 디바이스로 주당 5회 이상 리워드 요청 불가 

### 처리율 제한 장치를 사용하는 이유 
#### 1)Dos 공격 방지 
-  자원 고갈(resource starvation)방지 
-  임계치를 넘는 요청 차단 

예시 
- 트위터: 3시간 동안 300 트위 제한 
- 구글 Docs: 사용자당 분당 300 read 요청 제한 

-> 초과 요청을 차단하여 Dos 공격을 방지 

#### 2)비용 절감 
- 추가 요청을 제한 -> 서버 수 줄일 수 있음 
- 중요한 API에 자원 집중 가능 
- 특히 third-party API 사용 시 중요 (요청 횟수 기반 과금 구조)

#### 3) 서버 과부화 방지 
- 봇 트래픽 차단 
- 잘못된 사용 패턴으로 발생하는 트래픽 제어 

## 1단계 : 문제 이해 및 설계 범위 확정 
처리율 제한 장치는 다양한 알고리즘으로 구현 가능 -> 각 알고리즘은 서로 다른 장단점을 가짐 

따라서 면접시 면접관과 소통하여 어떤 제한 장치를 설계할지 명확히 해야 한다.

### 6. 요구사항 정리 
#### 기능 요구사항 
- 설정된 처리율을 초과하는 요청은 정확하게 제한 

#### 비기능  요구사항 
1) 낮은 응답 시간 
- HTTP 응답 시간에 영향을 최소화해야 함 

2) 메모리 효율성
- 가능한 한 적은 메모리 사용 

3) 분산 처리 지원 
- 여러 서버/프로세스에서 공유 가능해야 함 

4) 예외 처리 
- 제한된 요청은 사용자에게 명확히 전달 

5) 높은 결함 감내성 (Fault Tolerance)
- 제한 장치 장애가 전체 시스템에 영향 주면 안됨 

## 2단계 : 개략적 설계 
- 처음부터 복잡하게 설계하지 않는다 
- 클라이언트-서버 모델 기반으로 시작 

**처리율 제한 장치를 어디에 둘 것인가?**

### 1. 클라이언트 측 
- 요청 위변조 가능 
- 모든 클라이언트 통제 어려움 

결론 : 신뢰할 수 없음 

### 2. 서버 측 
- 서버에서 직접 요청 제어 가능 

-> 일반적인 방식 

### 3. 미들웨어 방식
- API 서버 앞단에서 요청을 검사 

### 동장 예시 (초당 2 요청)
- 1,2 번째 요청 -> 통과
- 3번째 요청 -> 차단 

응답 -> HTTP 429(Too Many Requests)

### 4. API 게이트웨이 방식
- 마이크로서비스 환경에서 사용 

### 기능 
- Rate Limiting 
- 인증 
- SSL 처리 
- IP 관리 

처리율 제한 = 게이트웨이의 기능 중 하나 

### 어디에 둘 것인가? (결론)
정답 없음(상황 의존)

### 판단 기준 
- 기술 스택
- 알고리즘 선택 가능 여부 
- MSA 구조 여부 
- 개발 인력 / 시간 

### 처리율 제한 알고리즘 
### 1. 토큰 버킷(Token Bucket)
#### 동작 
- 일정 속도로 토큰 생성 
- 요청 1개 = 토큰 1개 사용 
- 토큰 없으면 요청 거부 
#### 파라미터 
- 버킷 크기 
- 토큰 생성 속도 
#### 특징 
- burst 트래픽 처리 가능 
- 구현 간단 
- 메모리 효율적 
#### 단점 
- 파라미터 튜닝 어려움 

### 2. 누출 버킷(Leaky Bucket)
#### 동작
- 요청을 큐에 저장 
- 일정 속도로 처리 
#### 특징 
- 일정한 처리 속도 유지 
- 안정적인 출력 
#### 단점 
- 트래픽 몰리면 요청 쌓임 
- 최신 요청 버려질 수 있음 

### 3. 고정 윈도 카운터 
#### 동작 
- 시간 구간별 카운트 
- 임계치 초과 시 요청 차단 
#### 특징 
- 단순 
- 메몰 효율적 
#### 단점 
- 윈도 경계에서 요청 폭증 가능 -> 실제 제한보다 더 많이 처리됨 

### 4. 이동 윈도 로그 
#### 동작 
- 요청 timestamp 저장
- 윈도 범위 내 요청 수 계산 
#### 특징 
- 매우 정확한 제한 
#### 단점 
- 메모리 많이 사용 

### 5. 이동 윈도 카운터 
#### 동작 
- 이전 + 현재 윈도 비율 기반 계산 
#### 특징
- burst 대응 가능
- 메모리 효율적 
#### 단점 
- 정확도 약간 낮음(근사치)

### 개략적 아키텍처 
- 요청 수를 카운트하고 한도 넘으면 차단 

#### 저장 위치
- DB(느림)
- 캐시(빠름) Ex) Redis

### Redis 사용 이유 
- 메모리 기반 -> 빠름 
- TTL 지원 

#### 주요 명령어 
- `INCR` -> 카운트 증가 
- `EXPIRE` -> 자동 삭제 

#### 전체 흐름 
1. 클라이언트 요청 
2. Rate Limiter (미들웨어)
3. Redis에서 카운터 조회

#### 분기 
- 한도 초과 -> 요청 거부 
- 한도 미만 -> 요청 전달 + 카운터 증가
